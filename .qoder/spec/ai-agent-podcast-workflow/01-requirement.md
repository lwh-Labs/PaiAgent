## 1. 项目概述与目标

### 1.1 项目定位

构建一个 **AI Agent 工作流编辑与执行平台**，核心产品形态为"Agent 流图执行面板"。用户无需编写代码，即可通过可视化拖拽方式编排 AI 工作流节点，完成调试与执行。

### 1.2 核心场景

**AI 播客场景**：

```
用户输入文字 → 大模型节点（LLM 处理） → 超拟人音频合成节点（CosyVoice TTS） → 结束节点（输出音频） → 调试面板播放音频
```

### 1.3 目标用户

单用户系统（无需登录认证），适用于个人开发者调试与演示 AI 工作流。

### 1.4 核心价值

- 可视化编排 AI 工作流，降低 AI 应用搭建门槛
- 支持多厂商 LLM API（OpenAI、DeepSeek、通义千问）
- 集成超拟人 TTS（CosyVoice），支持音频合成与播放
- 调试模式实时查看每个节点执行状态与输出

## 3. 前端功能需求

### 3.1 左侧节点面板

**功能描述**：提供可拖拽的节点分类列表，用户从面板拖入节点到画板区域。

| 分类 | 节点类型 | 说明 |
|------|----------|------|
| 大模型节点 | LLM 调用节点 | 支持多厂商 LLM API |
| 工具节点 | 超拟人音频合成节点 | CosyVoice TTS |

**验收标准**：
- 节点以卡片形式展示，含图标与名称
- 支持拖拽到画板区域并自动生成对应节点
- 面板样式与整体 UI 风格统一

### 3.2 中央画板区域（React Flow）

**功能描述**：可视化流图编辑区域，基于 React Flow 实现节点拖拽、连线、删除等操作。

#### 3.2.1 画板基础操作

| 操作 | 描述 |
|------|------|
| 拖拽移动节点 | 节点可在画板内自由移动 |
| 节点连线 | 拖拽节点端口建立数据流向连线 |
| 删除节点/连线 | 选中后按 Delete 键或右键菜单删除 |
| 画板缩放/平移 | 支持鼠标滚轮缩放与拖拽平移 |
| 节点参数配置 | 点击节点打开配置面板（侧边或弹窗） |

#### 3.2.2 节点类型（画板内）

| 节点类型 | 标识 | 端口 | 角色 |
|----------|------|------|------|
| 用户输入节点 | START | 仅输出端口 | 工作流起始节点 |
| 大模型节点 | LLM | 输入端口 + 输出端口 | 调用 LLM API 处理文本 |
| 超拟人音频合成节点 | TTS | 输入端口 + 输出端口 | 调用 CosyVoice 合成语音 |
| 结束节点 | END | 仅输入端口 | 工作流终止节点 |

#### 3.2.3 节点视觉状态

| 状态 | 视觉表现 |
|------|----------|
| 默认 | 正常样式 |
| 运行中 | 蓝色边框 + loading 动效 |
| 执行成功 | 绿色边框 |
| 执行失败 | 红色边框 + 错误提示 |

**验收标准**：
- 画板可正常保存当前流图布局（节点位置、连线关系、节点参数）
- 工作流保存到后端 MySQL
- 页面刷新后可恢复上次保存的工作流
- 节点状态在调试执行时实时更新

### 3.3 调试抽屉（Debug Drawer）

**功能描述**：位于界面底部或右侧可展开的调试面板，用于输入测试数据、执行工作流、查看执行过程与结果。

#### 3.3.1 调试面板功能列表

| 功能 | 描述 |
|------|------|
| 文字输入框 | 用户输入测试文本（对应用户输入节点的数据） |
| 执行按钮 | 触发工作流执行，调用后端 API |
| 执行状态展示 | 实时展示每个节点的执行状态（等待/运行中/成功/失败） |
| 节点输出展示 | 展示每个节点的输入与输出内容（可折叠） |
| 错误信息展示 | 执行失败时显示具体错误信息 |
| 音频播放器 | 工作流最终输出为音频时，提供内嵌音频播放控件 |

#### 3.3.2 音频播放

- 当工作流包含 TTS 节点且执行成功时，调试抽屉底部展示音频播放器
- 支持播放/暂停、进度条、音量控制
- 音频文件由后端返回 URL 地址

**验收标准**：
- 点击执行后，各节点状态在画板上同步更新
- 调试抽屉内按节点顺序展示每步的输入输出
- 音频合成成功后自动展示播放器，可正常播放音频

### 3.4 交互逻辑

- 工作流编辑完成后，点击"保存"将流图配置持久化到后端
- 点击"调试执行"打开调试抽屉，触发工作流执行
- 执行过程中节点状态实时更新，不阻塞 UI 操作
- 执行完成后，调试抽屉展示完整执行记录

## 5. 节点类型详细定义

### 5.1 用户输入节点（START）

| 属性 | 描述 |
|------|------|
| **节点类型** | `START` |
| **角色** | 工作流起始节点，每个工作流有且仅有一个 |
| **输入** | 无（由用户在调试抽屉中输入） |
| **输出** | `{ "text": "用户输入的文字内容" }` |
| **配置项** | 无特殊配置项 |
| **约束** | 必须是 DAG 的唯一起始节点，无入边 |

### 5.2 大模型节点（LLM）

| 属性 | 描述 |
|------|------|
| **节点类型** | `LLM` |
| **角色** | 调用 LLM API 对输入文本进行处理 |
| **输入** | `{ "text": "来自上游节点的文本" }` |
| **输出** | `{ "text": "LLM 生成的文本" }` |
| **配置项** | 见下表 |

**LLM 节点配置项**：

| 配置项 | 类型 | 必填 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `provider` | enum | 是 | `openai` | LLM 提供商：`openai` / `deepseek` / `qwen` |
| `model` | string | 是 | - | 模型名称，如 `gpt-4o`、`deepseek-chat`、`qwen-turbo` |
| `apiKey` | string | 是 | - | API 密钥（前端不展示明文，后端存储） |
| `systemPrompt` | string | 否 | - | 系统提示词 |
| `temperature` | float | 否 | `0.7` | 生成温度，范围 0~2 |
| `maxTokens` | int | 否 | `2048` | 最大 token 数 |

### 5.3 超拟人音频合成节点（TTS）

| 属性 | 描述 |
|------|------|
| **节点类型** | `TTS` |
| **角色** | 调用 CosyVoice API，将文本转为高质量拟人语音 |
| **输入** | `{ "text": "待合成的文本内容" }` |
| **输出** | `{ "audioUrl": "/api/audio/xxx.mp3", "duration": 12.5 }` |
| **配置项** | 见下表 |

**TTS 节点配置项**：

| 配置项 | 类型 | 必填 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `apiKey` | string | 是 | - | CosyVoice API 密钥 |
| `voiceId` | string | 是 | - | 音色 ID，如 `longxiaochun` |
| `speed` | float | 否 | `1.0` | 语速，范围 0.5~2.0 |
| `pitch` | float | 否 | `1.0` | 音调，范围 0.5~2.0 |

### 5.4 结束节点（END）

| 属性 | 描述 |
|------|------|
| **节点类型** | `END` |
| **角色** | 工作流终止节点，收集最终输出 |
| **输入** | 来自上游节点的任意数据 |
| **输出** | 将上游节点输出作为最终结果返回给调试面板 |
| **配置项** | 无特殊配置项 |
| **约束** | 至少有一个入边，无出边 |

## 6. 工作流引擎需求

### 6.1 引擎设计原则

- 自研轻量级 DAG 执行引擎，不依赖第三方工作流框架
- 基于拓扑排序（Kahn 算法）确定节点执行顺序
- 节点间通过上下文对象（ExecutionContext）传递数据
- 支持串行执行（本期不需要并行分支）

### 6.2 核心执行流程

```
1. 加载工作流配置（节点列表 + 连线列表）
2. 构建 DAG 图（邻接表表示）
3. 执行拓扑排序，生成节点执行序列
4. 校验 DAG 合法性（无环、起始节点唯一、终止节点至少一个）
5. 按拓扑顺序依次执行各节点：
   a. 从 ExecutionContext 中获取上游节点输出作为本节点输入
   b. 调用节点对应的 Handler（LLMHandler / TTSHandler 等）
   c. 将本节点输出写入 ExecutionContext
   d. 更新节点执行状态（RUNNING → SUCCESS / FAILED）
6. 执行完成后，返回完整执行结果
```

### 6.3 执行状态管理

| 状态 | 描述 |
|------|------|
| `PENDING` | 等待执行 |
| `RUNNING` | 执行中 |
| `SUCCESS` | 执行成功 |
| `FAILED` | 执行失败（含错误信息） |
| `SKIPPED` | 跳过（上游节点失败时） |

### 6.4 异常处理策略

- 任意节点执行失败时，**终止后续节点执行**，工作流状态标记为 `FAILED`
- 记录失败节点的错误信息（异常类型、错误消息、堆栈）
- 错误信息通过 API 响应返回前端，调试抽屉展示具体错误

### 6.5 数据传递规范

- 节点间数据通过 `ExecutionContext` 对象传递
- Context 以 `Map<nodeId, nodeOutput>` 形式存储每个节点的输出
- 节点读取自身所有上游节点的输出，上游有多个时合并数据

## 7. 大模型集成需求

### 7.1 支持的 LLM 提供商

| 提供商 | API 协议 | 支持模型示例 |
|--------|----------|-------------|
| OpenAI | OpenAI Chat Completions API | `gpt-4o`, `gpt-4o-mini`, `gpt-3.5-turbo` |
| DeepSeek | OpenAI 兼容接口 | `deepseek-chat`, `deepseek-reasoner` |
| 通义千问（Qwen） | 阿里云 DashScope API | `qwen-turbo`, `qwen-plus`, `qwen-max` |

### 7.2 集成方案

- 使用**策略模式**封装不同 LLM 提供商的 API 调用差异
- 定义统一的 `LLMProvider` 接口，各提供商实现该接口
- DeepSeek 兼容 OpenAI 接口，通过修改 `baseUrl` 复用 OpenAI 客户端

### 7.3 请求参数规范

| 参数 | 描述 | 示例 |
|------|------|------|
| `model` | 模型名称 | `gpt-4o` |
| `messages` | 消息列表，含 system + user | `[{role: "user", content: "..."}]` |
| `temperature` | 生成温度 | `0.7` |
| `max_tokens` | 最大输出 token 数 | `2048` |

### 7.4 API 密钥管理

- API 密钥存储在后端数据库（节点配置中），**不在前端明文展示**
- 前端配置 API 密钥时，提交后端加密存储，显示为 `****` 形式

## 8. TTS 集成需求（CosyVoice）

### 8.1 CosyVoice 简介

CosyVoice 是通义实验室推出的超拟人语音合成服务，支持高质量语音合成，具备多音色、情感控制等能力。

### 8.2 集成方案

- 调用 **阿里云 DashScope CosyVoice API**（HTTP 接口）
- 输入：文本内容 + 音色 ID + 语速/音调参数
- 输出：音频文件（MP3/WAV 格式）

### 8.3 音频文件处理

- 后端接收 CosyVoice 返回的音频数据
- 将音频文件**临时存储**到本地文件系统（`/tmp/audio/` 目录）
- 通过 `/api/audio/{filename}` 接口提供音频文件访问
- 前端调试抽屉通过返回的 URL 加载音频播放器

### 8.4 超时与重试

- CosyVoice API 调用超时设置：**60 秒**
- 失败时不自动重试，直接返回错误状态，由用户手动重新执行

## 9. 调试功能需求

### 9.1 调试抽屉交互流程

```
1. 用户在调试抽屉输入测试文字
2. 点击"执行"按钮
3. 前端调用 POST /api/workflows/{id}/execute 接口
4. 后端同步执行工作流（本期不需要 WebSocket 实时推送，等待接口响应）
5. 接口返回完整执行结果（含各节点状态和输出）
6. 前端逐节点展示执行状态（解析响应数据渲染 UI）
7. 若最终输出包含音频 URL，展示音频播放器
```

> **说明**：本期采用**同步 HTTP 请求**方式，不使用 WebSocket。执行期间前端展示 loading 状态，等待接口响应后一次性渲染所有节点结果。

### 9.2 节点执行结果展示

每个节点在调试抽屉中展示：
- 节点名称与类型图标
- 执行状态标签（成功/失败/运行中）
- 执行耗时
- 输入数据（可折叠展示）
- 输出数据（可折叠展示）
- 失败时的错误信息

### 9.3 音频播放器

- 使用 HTML5 原生 `<audio>` 标签或成熟的 React 音频播放组件
- 支持：播放/暂停、进度拖拽、音量调节
- 音频加载失败时显示错误提示

## 10. 数据存储需求

### 10.1 数据库：MySQL

数据库名：`ai_agent_workflow`

### 10.2 表结构概要

#### 表 1：`workflow`（工作流配置表）

| 字段 | 类型 | 描述 |
|------|------|------|
| `id` | BIGINT PK | 主键，自增 |
| `name` | VARCHAR(255) | 工作流名称 |
| `description` | TEXT | 工作流描述 |
| `graph_json` | LONGTEXT | 工作流图配置（节点+连线+参数的 JSON） |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 最后更新时间 |

`graph_json` 字段示例结构：
```json
{
  "nodes": [
    { "id": "node_1", "type": "START", "position": { "x": 100, "y": 200 }, "config": {} },

    { "id": "node_3", "type": "TTS", "position": { "x": 500, "y": 200 }, "config": { "voiceId": "longxiaochun" } },
    { "id": "node_4", "type": "END", "position": { "x": 700, "y": 200 }, "config": {} }
  ],
  "edges": [
    { "id": "edge_1", "source": "node_1", "target": "node_2" },
    { "id": "edge_2", "source": "node_2", "target": "node_3" },
    { "id": "edge_3", "source": "node_3", "target": "node_4" }
  ]
}
```

#### 表 2：`execution_record`（执行记录表）

| 字段 | 类型 | 描述 |
|------|------|------|
| `id` | BIGINT PK | 主键，自增 |
| `workflow_id` | BIGINT FK | 关联工作流 ID |
| `status` | VARCHAR(20) | 执行状态：`RUNNING` / `SUCCESS` / `FAILED` |
| `input_json` | TEXT | 用户输入数据 JSON |

## 12. 非功能性需求

### 12.1 性能

- 工作流保存接口响应时间 < 500ms
- LLM 节点执行超时上限：120 秒
- TTS 节点执行超时上限：60 秒
- 前端画板支持 20 个节点以内的流图流畅渲染

### 12.2 可靠性

- 工作流配置不因页面刷新丢失（持久化到 MySQL）
- LLM/TTS 调用失败时，返回明确错误信息而非静默失败
- 音频文件临时存储，不做长期保留（可接受）

### 12.3 可维护性

- 节点 Handler 采用策略模式，便于后续新增节点类型
- LLM 提供商采用接口抽象，便于新增厂商支持
- 前端节点组件化，每种节点类型独立封装为 React 组件

### 12.4 安全性

- API 密钥不在前端响应中明文返回（展示为 `****`）
- 无需用户认证（单用户系统）
- 音频文件访问无需鉴权

### 12.5 开发阶段约束

- 本期为 MVP 版本，不需要多用户支持、权限管理
- 不需要工作流版本管理
- 不需要节点并行执行（仅串行 DAG）
- 不需要 WebSocket 实时推送（同步 HTTP 即可）

## 13. 复杂度评估

**复杂度**：高

**评估依据**：

| 维度 | 评估 |
|------|------|
| 功能点数量 | 10+ 个功能点，涵盖前端、后端、引擎、集成多个层次 |
| 涉及范围 | 跨模块：前端流图编辑器 + 后端工作流引擎 + LLM 集成 + TTS 集成 |
| 技术难度 | 需要自研 DAG 引擎、集成多厂商 API、React Flow 可视化编排 |
| 一次性完成信心 | 中等 - 核心需求清晰，但工作流引擎与外部 API 集成存在一定实现复杂度 |

**建议**：进入高层设计（HLD）阶段，分模块进行架构设计，再逐模块实现，降低返工风险。建议实现顺序：

1. **后端基础框架** → 数据库 + API 骨架
2. **DAG 工作流引擎** → 核心执行逻辑
3. **LLM 节点集成** → 通 OpenAI，再扩展其他厂商
4. **TTS 节点集成** → CosyVoice 对接
5. **前端画板** → React Flow 基础画板
6. **前端节点配置** → 各节点配置面板
7. **调试抽屉** → 执行流程 + 音频播放
8. **联调测试** → 端到端 AI 播客场景验证
